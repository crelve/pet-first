# implementation-guard

## 概要
実装作業の品質を保証するためのガードコマンドです。すべての実装系コマンドで品質チェックが確実に実行されるようにガイドします。

## 用途
- 実装完了時の品質チェック忘れ防止
- PR作成前の必須チェック項目の確認
- チーム開発での品質基準統一

## 必須ワークフロー

### 1. 実装作業完了後
```bash
# 必須: 品質チェック実行
make check-ready
```

### 2. すべてクリア後
```bash
# 推奨: 安全なプッシュ
make push

# または手動でプッシュ
git push
```

### 3. PR作成時
- 品質チェック結果をPRに記載
- すべてのチェックが✅の状態でのみPR作成

## 品質チェック項目

### 自動チェック（make check-ready）
- ✅ **フォーマット**: `dart format lib/ test/`
- ✅ **静的解析**: `flutter analyze`（エラー0件必須）
- ✅ **テスト**: `flutter test`（全合格必須）

### 手動確認項目
- ✅ 実装が要件に合致している
- ✅ 既存機能に影響を与えていない
- ✅ 適切なコメント・ドキュメントが記述されている

## エラー対処フロー

### 1. 品質チェックでエラーが発生した場合
```bash
# 個別にチェックして問題を特定
make format     # フォーマットエラー
make analyze    # 静的解析エラー
make test       # テストエラー

# 問題修正後、再チェック
make check-ready
```

### 2. Lefthookでプッシュが拒否された場合
```bash
# 事前チェックで問題を解決
make check-ready

# 問題なければプッシュ
git push

# 緊急時のスキップ（非推奨）
LEFTHOOK=0 git push
```

## チーム開発ルール

### 実装者の責任
1. **実装完了時**: 必ず `make check-ready` を実行
2. **PR作成前**: すべてのチェックが✅であることを確認
3. **エラー対応**: 品質チェックでエラーが出た場合は修正してから提出

### レビュアーの確認事項
1. PR説明に品質チェック結果が記載されているか
2. CI/CDで品質チェックが通過しているか
3. 実装内容が品質基準を満たしているか

## 効果

### 開発効率向上
- **エラーの早期発見**: プッシュ前に問題を特定・修正
- **Lefthookエラー回避**: 品質チェック失敗によるプッシュ拒否を防止
- **レビュー時間短縮**: 品質が保証された状態でのレビュー

### コード品質向上
- **一貫性**: 統一されたフォーマット・スタイル
- **信頼性**: 静的解析とテストによる品質保証
- **保守性**: 適切な構造とドキュメント

## 関連コマンド
- `make check-ready`: 統合品質チェック
- `make push`: 品質チェック付きプッシュ
- `make format`: フォーマットのみ
- `make analyze`: 静的解析のみ
- `make test`: テストのみ
- `/fix-error`: エラー修正支援
- `/pr-create`: PR作成支援

## ベストプラクティス

### 推奨ワークフロー
```bash
# 1. 実装作業
# ... コーディング ...

# 2. 品質チェック（必須）
make check-ready

# 3. 問題があれば修正
# ... 修正作業 ...
make check-ready  # 再チェック

# 4. 安全なプッシュ
make push

# 5. PR作成
# 品質チェック結果を含めてPR作成
```

### 避けるべきパターン
```bash
# ❌ 品質チェックなしでプッシュ
git add . && git commit -m "fix" && git push

# ❌ エラーを無視してスキップ
LEFTHOOK=0 git push  # 緊急時以外は使用禁止

# ❌ 部分的なチェックのみ
make format  # これだけでは不十分
git push
```

このガードシステムにより、すべての実装作業で一定の品質が保証されます。

## Role Management

特定のロール（役割）に切り替えて専門的な分析や作業を実行し、複数ロール間での議論や比較分析も可能な包括的なロール管理システム。

### 使い方

```bash
# 基本的なロール実行
/role <ロール名> [--agent|-a]

# 複数ロール並行分析
/role multi <ロール 1>,<ロール 2> [--agent|-a] [分析対象]

# ロール間議論
/role debate <ロール 1>,<ロール 2> [議題]

# ロール選択ヘルプ
/role help [状況/問題]
/role help compare <ロール 1>,<ロール 2>
```

### オプション

- `--agent` または `-a` : サブエージェントとして独立実行（大規模分析時推奨）
  - このオプションを使用すると、ロールの description に自動委任促進フレーズ（"use PROACTIVELY" など）が含まれている場合、より積極的な自動委任が有効になります

### 利用可能なロール

#### 専門分析ロール（ Evidence-First 統合）

- `security` : セキュリティ監査専門家（ OWASP Top 10 ・脅威モデリング・ Zero Trust 原則・ CVE 照合）
- `performance` : パフォーマンス最適化専門家（ Core Web Vitals ・ RAIL モデル・段階的最適化・ ROI 分析）
- `analyzer` : 根本原因分析専門家（ 5 Whys ・システム思考・仮説駆動・認知バイアス対策）
- `frontend` : フロントエンド・ UI/UX 専門家（ WCAG 2.1 ・デザインシステム・ユーザー中心設計）

#### 開発支援ロール

- `reviewer` : コードレビュー専門家（可読性・保守性・パフォーマンス・リファクタリング提案）
- `architect` : システムアーキテクト（ Evidence-First 設計・ MECE 分析・進化的アーキテクチャ）
- `qa` : テストエンジニア（テストカバレッジ・ E2E/統合/単体戦略・自動化提案）
- `mobile` : モバイル開発専門家（ iOS HIG ・クロスプラットフォーム戦略）

#### Flutter/Dart 特化ロール

- `flutter` : Flutter/Dart ベストプラクティス専門家（ Widget 設計・状態管理・パフォーマンス最適化）
- `test` : テスト戦略と品質保証専門家（ユニット・ Widget ・統合テストの最適化と TDD 実践支援）

## 複数ロール並行分析

複数のロールで同じ対象を並行分析し、統合レポートを生成する機能。

### 使い方

```bash
/role multi <ロール 1>,<ロール 2> [--agent|-a] [分析対象]
/role multi <ロール 1>,<ロール 2>,<ロール 3> [--agent|-a] [分析対象]
```

**重要**:
- `--agent` オプションはロール指定の直後に配置してください
- メッセージは `--agent` の後に記述してください
- 正しい例: `/role multi qa,architect --agent 計画を評価して`
- 間違った例: `/role multi qa,architect 計画を評価して --agent`

### 基本例

```bash
# セキュリティとパフォーマンスの両面分析（通常）
/role multi security,performance
「この API エンドポイントを評価して」

# QA とアーキテクトによる設計評価（エージェント並列実行）
/role multi qa,architect --agent
「テスト戦略とアーキテクチャの観点から計画を評価して」

# 3 つの専門領域による包括評価
/role multi security,performance,analyzer --agent
「システム障害の原因を多角的に分析して」
```

### 並行分析の利点

- **多角的視点**: 異なる専門分野からの包括的評価
- **見落とし防止**: 単一視点では気づかない問題の発見
- **統合提案**: 複数の改善案を組み合わせた最適解
- **トレードオフ明確化**: 異なる優先度間のバランス検討

## ロール間議論

異なる専門性を持つロールが議論し、トレードオフを検討して最適解を導出する機能。

### 使い方

```bash
/role debate <ロール 1>,<ロール 2> [議題]
/role debate <ロール 1>,<ロール 2>,<ロール 3> [議題]
```

### 基本例

```bash
# セキュリティ vs パフォーマンスのトレードオフ
/role debate security,performance
「 JWT トークンの有効期限設定について」

# ユーザビリティ vs セキュリティのバランス
/role debate frontend,security
「 2 段階認証の UX 最適化について」

# 技術選定の議論
/role debate architect,mobile
「 React Native vs Flutter の選択について」

# 3 者議論
/role debate architect,security,performance
「マイクロサービス化の是非について」
```

### 議論の流れ

1. **問題提起**: 各ロールが自分の専門分野から課題を特定
2. **対立点明確化**: 異なる優先度や判断基準を明示
3. **トレードオフ検討**: 各選択肢のメリット・デメリットを議論
4. **合意形成**: 最適なバランスポイントの特定
5. **統合提案**: 全体最適な解決策の提示

## ロール選択ヘルプ

どのロールを使うべきか迷った時の選択ガイドとヘルプシステム。

### 使い方

```bash
/role help                         # 全般的なロール選択ガイド
/role help <状況/問題>             # 特定状況での推奨ロール
/role help compare <ロール 1>,<ロール 2> # ロール比較
```

### 迷った時のフローチャート

```
問題の性質は？
├─ セキュリティ関連 → security
├─ パフォーマンス問題 → performance  
├─ バグ・障害調査 → analyzer
├─ UI/UX 改善 → frontend or mobile
├─ 設計・アーキテクチャ → architect
├─ コード品質 → reviewer
├─ テスト関連 → qa
└─ 複合的・複雑 → multi で複数ロール分析

複数の分野にまたがる？
├─ 統合分析したい → multi
├─ 議論・トレードオフ → debate
└─ 判断に迷う → help で相談
```

### ロール比較表

#### セキュリティ系
| ロール | 主な用途 | 得意分野 | 苦手分野 |
|--------|----------|----------|----------|
| security | 脆弱性・攻撃対策 | 脅威分析、認証設計 | UX 、パフォーマンス |
| analyzer | 根本原因分析 | 論理的分析、証拠収集 | 予防策、将来計画 |

#### 設計系
| ロール | 主な用途 | 得意分野 | 苦手分野 |
|--------|----------|----------|----------|
| architect | システム設計 | 長期視点、全体最適 | 詳細実装、短期解決 |
| reviewer | コード品質 | 実装レベル、保守性 | ビジネス要件、 UX |

#### パフォーマンス系
| ロール | 主な用途 | 得意分野 | 苦手分野 |
|--------|----------|----------|----------|
| performance | 高速化・最適化 | 測定、ボトルネック | セキュリティ、 UX |
| qa | 品質保証 | テスト、自動化 | 設計、アーキテクチャ |

#### ユーザー体験系
| ロール | 主な用途 | 得意分野 | 苦手分野 |
|--------|----------|----------|----------|
| frontend | Web UI/UX | ブラウザ、アクセシビリティ | サーバーサイド、 DB |
| mobile | モバイル UX | タッチ、オフライン対応 | サーバーサイド、 Web |

### よくある質問

#### Q: frontend と mobile の違いは？
```
A: 
frontend: Web ブラウザ中心、 HTML/CSS/JavaScript
mobile: モバイルアプリ中心、 iOS ネイティブ・ React Native など

両方関連する場合は /role multi frontend,mobile がおすすめ
```

#### Q: security と analyzer の使い分けは？
```
A:
security: 攻撃・脅威の予防、セキュリティ設計
analyzer: 既に起きた問題の原因分析、調査

セキュリティインシデントの調査なら /role multi security,analyzer
```

#### Q: architect と performance の違いは？
```
A:
architect: システム全体の長期的設計、拡張性
performance: 具体的な速度・効率の改善

大規模システムの性能設計なら /role multi architect,performance
```

## 実用例

### 複合的な問題への対応

```bash
# 例 1: ログイン機能の問題
/role multi security,frontend,performance
「ログイン画面が遅く、ユーザビリティが悪いという報告がある」

# 例 2: システム設計の議論  
/role debate architect,security,performance
「新しい決済システムの技術選定について議論して」

# 例 3: 品質改善の包括検討
/role multi qa,reviewer,performance
「リリース前の品質チェックリストを作成して」
```

### 段階的な問題解決

```bash
# ステップ 1: 問題の特定
/role analyzer
「システムが不安定になる原因を分析して」

# ステップ 2: 解決策の検討
/role multi architect,performance,security
「分析結果に基づいて改善案を提案して」

# ステップ 3: 実装の議論
/role debate qa,reviewer
「提案された改善案の実装優先度を決めて」
```

## 注意事項

- **複雑な問題ほど複数ロールの組み合わせが効果的**
- **緊急性が高い場合は単一ロールで迅速に対応**
- **迷った時は help で相談してから決定**
- **最終的な判断はユーザーが問題の性質を考慮して決定**
- **エージェントモード（--agent ）は大規模分析時に活用**

# 🔄 リテンション分析コマンド

<!-- LANGUAGE: ja -->
<!-- RESPONSE_LANGUAGE: Japanese -->

**ユーザーリテンション（継続率）を深堀り分析し、改善施策を提案**

## 🎯 概要

このコマンドは、Firebase Analytics のリテンションデータを分析し、以下を実行します:
- コホート分析（日別・週別・月別）
- リテンション低下ポイントの特定
- ユーザーセグメント別リテンション比較
- チャーン（離脱）要因分析
- リテンション改善施策の優先順位付け

**IMPORTANT:** このコマンド実行前に、必ず以下を参照してください:
- **[トークン効率化ベストプラクティス](../../docs/best-practices/token-efficiency.md)** - 10個の黄金ルール

---

## 📊 リテンション分析の重要性

```
「なぜリテンションが最重要指標なのか:

💰 収益への影響:
   - 新規ユーザー獲得コスト: $1-5
   - 既存ユーザー維持コスト: $0.10-0.50
   - LTV の 80% は継続利用から発生

📈 成長への影響:
   - 5% のリテンション改善 = 25-95% の利益向上
   - バイラル成長の前提条件（K-factor > 1）
   - 高リテンション = 低 CAC = 持続可能な成長

🎯 業界ベンチマーク:
   - D1（1日後）: 40-50% が優秀
   - D7（7日後）: 20-30% が優秀
   - D30（30日後）: 10-20% が優秀
   - モバイルゲーム平均: D1=25%, D7=10%, D30=5%」
```

### 1. コホート分析

#### 1.1 日別コホート分析

```bash
「各インストール日のユーザーグループを追跡:

📊 分析軸:

💡 洞察例:
   - 「月曜インストールユーザーは D7 が 10% 高い」
```

### 3. チャーン（離脱）要因分析

#### 3.1 離脱タイミング分析

```bash
「いつユーザーが離脱するか:

📉 Critical Drop-off Points:
   - D0-D1: 初日体験の問題
   - D1-D3: 価値提供不足
   - D7-D14: 習慣化失敗
   - D30+: 競合への流出

🔍 セッション単位の離脱:
   - 1回起動のみで離脱: XX%
   - 2-5回で離脱: XX%
   - 6-10回で離脱: XX%
   - 30回以上で定着: XX%

💡 洞察例:
   - 「60% が初回セッションのみで離脱」
   - 「D3 で急激な 30% ドロップ」
   - 「10回以上起動すれば 80% が継続」」
```

#### 3.2 離脱ユーザーの行動パターン

```bash
「離脱前にユーザーが何をしていたか:

📱 最終セッションの特徴:

   - 最後に使った機能
   - エラー・クラッシュ発生の有無
   - 課金トライアル失敗の有無

🚫 離脱のトリガーイベント:
   - 広告表示直後の離脱
   - アップデート直後の離脱
   - 課金ダイアログ表示後の離脱
   - 競合アプリインストール後の離脱

💡 洞察例:
   - 「インタースティシャル広告後 20% が即離脱」
   - 「課金促進後の D1 が 15% 低下」
   - 「エラー体験ユーザーの 70% が離脱」」
```

### 4. リテンション改善施策

#### 4.1 優先度マトリクス（Impact vs Effort）

```bash
「施策の優先順位付け:

🚀 Quick Wins（高インパクト・低工数）:
   □ プッシュ通知最適化（リエンゲージメント）
   □ オンボーディング改善（Aha! モーメント早期化）
   □ D3-D7 リマインダー通知追加
   □ 初回課金インセンティブ追加

💎 Big Bets（高インパクト・高工数）:
   □ ゲーミフィケーション導入（デイリーチャレンジ）
   □ ソーシャル機能追加（友達招待）
   □ パーソナライゼーション強化（AI推薦）
   □ サブスクリプションモデル再設計

⚡ Fill Ins（低インパクト・低工数）:
   □ アプリアイコンバッジ通知
   □ メール再エンゲージメント
   □ アプリ内レビュー依頼最適化
   □ ウィジェット追加

❌ Thankless Tasks（低インパクト・高工数）:
   □ 複雑な新機能追加（利用率不明）
   □ 大規模UI刷新
   □ 過度なアニメーション追加」
```

#### 4.2 オンボーディング最適化

```bash
「初回体験で "Aha! Moment" を早期化:

🎯 目標:
   - D1 リテンション: 現状 XX% → 目標 +10%
   - オンボーディング完了率: 現状 XX% → 目標 80%
   - 初回セッション時間: 現状 XX秒 → 目標 3分

✅ 改善施策:
   □ チュートリアル簡素化（5ステップ → 3ステップ）
   □ プログレスバー追加（完了までの視覚化）
   □ スキップ可能化（経験者向け）
   □ 価値提案の明確化（"なぜこれが便利か"）
   □ インタラクティブチュートリアル（読むだけ → 実際に操作）

📊 測定指標:
   - オンボーディング各ステップの離脱率
   - 完了までの平均時間
   - スキップ率
   - 完了ユーザーの D7 リテンション」
```

#### 4.3 プッシュ通知戦略

```bash
「リエンゲージメント通知の最適化:

📢 通知タイプ別戦略:

1. D1 リマインダー（インストール後 24時間）:
   「昨日は [アプリ名] をお試しいただきありがとうございます！
    今日も [主要機能] を使ってみませんか？」
   - 送信タイミング: 前日と同じ時間帯
   - 期待効果: D1 → D2 リテンション +5-10%

2. D3 リエンゲージメント（3日間未起動）:
   「[ユーザー名] さん、[新機能/お得情報] をお見逃しなく！」
   - パーソナライズ: 過去の利用パターンに基づく
   - 期待効果: D3 → D7 リテンション +10-15%

3. D7 ハビットリマインダー（7日間未起動）:
   「[アプリ名] が恋しいです😊 戻ってきませんか？」
   - ディープリンク: 最後に使った機能へ誘導
   - 期待効果: 休眠ユーザー復活率 +20%

4. 価値実感通知（定期的）:
   「今月あなたは [数値] を達成しました！」
   - 実績の可視化: 使った時間、節約した金額等
   - 期待効果: 長期リテンション +15%

⏰ 送信タイミング最適化:
   - ユーザーの過去アクティブ時間帯に送信
   - タイムゾーン考慮
   - 深夜・早朝は避ける（離脱原因になる）
   - A/B テストで最適化

🚫 注意点:
   - 過剰送信は逆効果（週1-2回まで）
   - 通知オプトアウト率を監視（5% 以下が目標）
   - 通知クリック率を測定（10% 以上が目標）」
```

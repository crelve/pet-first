# 🎯 コンバージョンファネル分析コマンド

<!-- LANGUAGE: ja -->
<!-- RESPONSE_LANGUAGE: Japanese -->

**ユーザージャーニー全体を可視化し、離脱ポイントを特定・改善**

## 🎯 概要

このコマンドは、ストア訪問から課金まで全体を分析し、以下を実行します:
- ファネル全体の可視化（各ステップのCVR）
- 最大の離脱ポイントの特定
- セグメント別ファネル比較
- 改善施策の優先順位付け
- A/Bテスト計画の策定

**IMPORTANT:** このコマンド実行前に、必ず以下を参照してください:
- **[トークン効率化ベストプラクティス](../../docs/best-practices/token-efficiency.md)** - 10個の黄金ルール

---

## 📊 コンバージョンファネルとは

```
「ユーザーが辿る道筋とそのCVR:

📱 典型的なモバイルアプリファネル:

1. ストアページ訪問 ────────── 100% (1,000人)
   ↓ CVR: 30%
2. インストール ──────────── 30% (300人)
   ↓ CVR: 40%
3. 初回起動 ────────────── 12% (120人)
   ↓ CVR: 50%
4. オンボーディング完了 ────── 6% (60人)
   ↓ CVR: 20%
5. 主要機能利用 ────────────  1.2% (12人)
   ↓ CVR: 10%
6. 課金 ────────────────── 0.12% (1.2人)

💡 分析のポイント:
   - 最大の離脱ポイントはどこか?
   - 業界平均と比べてどうか?
   - セグメント別に傾向は異なるか?
   - どのステップを改善すべきか?

🎯 業界ベンチマーク:
   - ストア→インストール: 20-40%
   - インストール→初回起動: 60-80%
   - 初回起動→オンボーディング完了: 30-60%
   - オンボーディング→主要機能利用: 20-40%
   - 初回起動→課金: 1-5%」
```

## 📋 実行内容

### 1. ファネル定義

#### 1.1 アクイジションファネル（獲得）

```bash
「ユーザーがアプリを見つけてインストールするまで:

📊 ステップ:

1. インプレッション（ストアでの露出）
   - 検索結果表示
   - カテゴリランキング表示
   - おすすめ表示

2. ストアページ訪問
   - 測定: ストアページビュー数
   - ソース別: オーガニック/ペイド広告/リファラル

3. スクリーンショット閲覧
   - 何枚目まで見たか
   - プレビュー動画の視聴率

4. レビュー閲覧
   - レビューセクションまでスクロール
   - 高評価/低評価の閲覧

5. インストールボタンタップ
   - CVR: ページビュー → タップ
   - ソース別CVR比較

6. インストール完了
   - CVR: タップ → 完了
   - キャンセル率の測定

💡 改善施策:
   - ASO最適化（検索順位向上）
   - スクリーンショット改善（CVR向上）
   - レビュー評価向上（信頼性向上）」
```

#### 1.2 アクティベーションファネル（有効化）

```bash
「インストール後、ユーザーが価値を実感するまで:

📊 ステップ:

1. 初回起動
   - インストール → 初回起動率: 60-80%
   - 起動しないユーザーの要因分析

2. パーミッション許可
   - プッシュ通知: 30-50%
   - 位置情報: 10-30%
   - カメラ・写真: 状況により

3. アカウント作成/ログイン
   - ソーシャルログイン: CVR 60-80%
   - メールアドレス登録: CVR 20-40%
   - スキップ（ゲスト）: CVR 10-20%

4. オンボーディングチュートリアル
   - 開始率
   - 各ステップの離脱率
   - 完了率: 30-60%

5. 初回コア機能利用（"Aha! Moment"）
   - 主要機能の初回利用
   - 価値実感までの時間
   - CVR: オンボーディング完了 → コア機能利用

💡 改善施策:
   - オンボーディング簡素化
   - パーミッション依頼タイミング最適化
   - ソーシャルログイン導入
   - "Aha! Moment" の早期化」
```

#### 1.3 エンゲージメントファネル（継続利用）

```bash
「ユーザーがアプリを習慣化するまで:

📊 ステップ:

1. 2回目起動
   - D1（1日後）リターン率: 40-50%
   - 理由: 価値を実感できたか

2. 3-5回目起動
   - 習慣化の萌芽
   - 週次利用パターンの確立

3. 主要機能の定期利用
   - デイリーアクティブユーザー（DAU）化
   - ウィークリーアクティブユーザー（WAU）化

4. 二次機能の利用
   - 機能の幅を広げる
   - アプリ内滞在時間の増加

5. パワーユーザー化
   - 月10回以上利用
   - 複数機能を組み合わせ利用

💡 改善施策:
   - プッシュ通知でリエンゲージメント
   - ゲーミフィケーション導入
   - 二次機能への誘導強化」
```

#### 1.4 マネタイゼーションファネル（収益化）

```bash
「ユーザーが課金するまで:

📊 ステップ:

1. 有料機能の認知
   - 課金誘導バナー表示
   - 無料枠の上限到達

2. 課金ページ訪問
   - CVR: アプリ利用 → 課金ページ訪問
   - ソース: 自発的 vs プロモーション経由

3. プラン選択
   - 月額 vs 年額
   - 無料トライアル開始

4. 決済情報入力
   - CVR: プラン選択 → 決済完了
   - 離脱理由: 価格、決済手段等

5. 初回課金完了
   - CVR: 初回起動 → 初回課金 (1-5%)
   - 課金までの平均日数

6. 継続課金（サブスク）
   - 月次継続率: 60-80%
   - 年次継続率: 30-50%
   - チャーン率の測定

💡 改善施策:
   - 無料トライアル提供
   - 価格帯の最適化
   - 決済手段の拡充
   - リテンションベースの価格設定」
```

```bash

1. サンキーダイアグラム:
   各ステップ間の流れを視覚化
   離脱の大きさが一目瞭然

2. ファネルチャート:
   各ステップのCVRを縦積み表示

💡 ツール:
```

### 3. セグメント別分析

```bash
「どのユーザーセグメントがコンバートしやすいか:

📊 分析軸:

1. 流入元別:
   - オーガニック検索
   - Apple Search Ads
   - Google App Campaign
   - Facebook Ads
   - リファラル

   💡 洞察例:
   「Apple Search Ads 経由は
    インストールCVRは低いが
    課金CVRは3倍高い」

2. デバイス別:
   - iOS vs Android
   - デバイスモデル
   - OS バージョン

   💡 洞察例:
   「iOS ユーザーの課金CVRは
    Android の 2倍」

3. 地域別:
   - 国/地域
   - 言語設定

   💡 洞察例:
   「日本ユーザーのARPUは
    米国の 1.5倍」

4. 行動パターン別:
   - 初回セッション時間
   - オンボーディング完了 vs スキップ
   - 通知許可 vs 拒否

   💡 洞察例:
   「プッシュ通知許可ユーザーの
    D7リテンションは3倍、
    課金CVRは5倍」

5. 時間軸別:
   - 曜日（月〜日）
   - 時間帯（0-23時）
   - 季節性

   💡 洞察例:
   「週末インストールユーザーは
    平日の 1.5倍エンゲージ」

📊 分析手法:
   - 各セグメントのファネル比較
   - ベスト/ワーストパフォーマー特定
   - パターンの一般化」
```

### 4. 離脱要因分析

```bash
「なぜユーザーは離脱するのか:

🔍 調査手法:

1. 定量分析:
   □ 離脱直前の行動ログ分析
   □ 離脱ユーザー vs 継続ユーザーの比較
   □ エラーログ・クラッシュレポート
   □ セッション録画（Hotjar等）

2. 定性分析:
   □ ユーザーレビューのテーマ抽出
   □ カスタマーサポート問い合わせ分析
   □ ユーザーインタビュー
   □ アンケート調査

📋 離脱要因の分類:

【技術的問題】
   - クラッシュ・バグ
   - パフォーマンス（重い、遅い）
   - 互換性問題（OS・デバイス）

【UX問題】
   - 使い方が分からない
   - オンボーディングが長すぎる
   - UI が複雑・直感的でない

【価値提供不足】
   - 期待と異なる
   - 機能が不十分
   - 競合の方が優れている

【課金・広告】
   - 価格が高すぎる
   - 広告が多すぎる・邪魔
   - 無料枠が狭すぎる

【その他】
   - ニーズの消失
   - 一時的な利用（イベント等）
   - 競合への乗り換え

💡 各要因への対策:
   - 技術的問題 → バグ修正、最適化
   - UX問題 → オンボーディング改善
   - 価値提供 → 機能拡充、差別化
   - 課金・広告 → 価格見直し、広告最適化」
```

### 5. 改善施策の優先順位付け

```bash
「Impact × Ease マトリクスで優先度決定:

🎯 評価軸:

Impact（インパクト）:
   - 影響を受けるユーザー数
   - CVR 改善の見込み幅
   - 収益への影響度
   - 5段階評価（1=低、5=高）

Ease（実装容易性）:
   - 開発工数
   - 必要なリソース
   - リスクの大きさ
   - 5段階評価（1=難、5=易）

📊 優先度マトリクス:

【Quick Wins（高Impact × 高Ease）】最優先
   例:
   □ オンボーディングのステップ削減
   □ プッシュ通知の文言改善
   □ 課金ページのCTAボタン最適化
   □ エラーメッセージの改善

【Major Projects（高Impact × 低Ease）】中期計画
   例:
   □ パーソナライゼーション機能追加
   □ ゲーミフィケーション導入
   □ 決済手段の拡充
   □ 機能の大幅刷新

【Fill Ins（低Impact × 高Ease）】余力があれば
   例:
   □ マイナーなUI調整
   □ ヘルプテキストの追加
   □ アニメーション追加

【Thankless Tasks（低Impact × 低Ease）】やらない
   例:
   □ 複雑な新機能（需要不明）
   □ 全面リニューアル

🚀 実行順序:
   1. Quick Wins を即実行（週次）
   2. Major Projects を計画的に（月次）
   3. Fill Ins は隙間時間で
   4. Thankless Tasks は避ける

📊 効果測定:
   - 各施策のBefore/After CVR
   - ROI 計算
   - 継続的な A/B テスト」
```

### 6. A/Bテスト計画

```bash
「ファネル改善のためのテストシナリオ:

🧪 テスト例:

【アクイジション】
Test 1: スクリーンショット
   - A: テキストオーバーレイなし
   - B: テキストオーバーレイあり
   - 測定: ストアページ訪問 → インストール CVR
   - 期待: +10-20%

【アクティベーション】
Test 2: オンボーディング
   - A: 5ステップ
   - B: 3ステップ（簡素化）
   - 測定: 開始 → 完了 CVR
   - 期待: +20-30%

Test 3: パーミッション依頼タイミング
   - A: 初回起動直後
   - B: 必要になったタイミング（JIT）
   - 測定: 許可率
   - 期待: +30-50%

【エンゲージメント】
Test 4: プッシュ通知文言
   - A: 機能中心（「新機能追加」）
   - B: ベネフィット中心（「あなたの目標達成まであと少し!」）
   - 測定: 通知クリック率、D7リテンション
   - 期待: +15-25%

【マネタイゼーション】
Test 5: 無料トライアル期間
   - A: 7日間無料
   - B: 14日間無料
   - 測定: トライアル開始率、有料転換率
   - 期待: 開始率+20%、転換率-5%（トレードオフ）

Test 6: 価格表示
   - A: 月額表示（¥980/月）
   - B: 日額換算（1日33円）
   - 測定: 課金CVR
   - 期待: +10-15%

💡 テスト設計のベストプラクティス:
   - 1度に1変数のみテスト
   - サンプルサイズ: 最低1000ユーザー/グループ
   - 統計的有意性: 95%信頼区間
   - テスト期間: 1-2週間（最低）
   - 事前に仮説・成功基準を明確化」
```

# 💬 Operations Phase 4: ユーザーフィードバック分析

## 目的
ユーザーの声を体系的に収集・分析し、プロダクト改善に活用する

---

## 前提条件
- App Store Connect / Google Play Consoleへのアクセス権限
- サポートメール受信環境
- （オプション）アプリ内フィードバック機能

---

## フィードバック収集チャネル

| チャネル | 収集頻度 | 優先度 | 内容 |
|---------|---------|--------|------|
| App Store / Play Store レビュー | 毎日 | ★★★ | 公開評価・要望 |
| サポートメール | 随時 | ★★★ | 詳細な問題報告 |
| アプリ内フィードバック | 随時 | ★★ | 機能要望・バグ報告 |
| SNS（Twitter等） | 週1回 | ★ | 評判・話題 |
| 競合アプリレビュー | 月1回 | ★ | 市場ニーズ把握 |

## Step 3: 機能要望管理

### 3.1 要望バックログ

| 要望 | リクエスト数 | 優先度 | 実装難度 | ステータス |
|------|-------------|--------|---------|-----------|
| | | | | 📋 検討中 / 🎯 予定 / 🚀 開発中 / ✅ 完了 |

### 3.2 優先度スコアリング（RICE）

```
RICE Score = (Reach × Impact × Confidence) / Effort

- Reach: 影響ユーザー数（推定）
- Impact: 影響度（3=大, 2=中, 1=小, 0.5=極小）
- Confidence: 確信度（100%, 80%, 50%）
- Effort: 工数（人日）
```

| 要望 | Reach | Impact | Confidence | Effort | RICE |
|------|-------|--------|------------|--------|------|
| | | | | | |

### 3.3 要望対応ポリシー

| リクエスト数 | 対応 |
|-------------|------|
| 10件以上 | 優先検討 |
| 5-9件 | 通常検討 |
| 1-4件 | バックログ保持 |

## Step 4: バグ・問題追跡

### 4.1 バグ報告の優先度分類

| 重要度 | 説明 | 対応期限 |
|--------|------|---------|
| 🔴 Critical | クラッシュ・データ消失 | 24時間以内 |
| 🟠 High | 主要機能の不具合 | 3日以内 |
| 🟡 Medium | 軽微な不具合 | 次回リリース |
| 🟢 Low | UI/UXの小さな問題 | バックログ |

## 🤖 Claude Code への指示

**トークン効率化:**
- レビューデータは要約して保存
- 重複フィードバックは統合
- バックログはRICEスコア順でソート

**実行モード:**
1. `--daily`: 新規レビュー確認・返信下書き
2. `--weekly`: 週次サマリー生成
3. `--monthly`: 月次分析レポート生成
4. `--analyze [期間]`: 指定期間のセンチメント分析

**自動化連携:**
- App Store Connect API / Google Play Developer API
- Slack / Discord への通知
- GitHub Issues との連携

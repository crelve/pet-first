# 🧪 Operations Phase 3: A/Bテスト管理

## 目的
データドリブンな意思決定でアプリの主要KPIを継続的に改善する

---

## 前提条件
- Firebase Remote Config が設定済み
- Firebase Analytics でイベントトラッキング設定済み
- 十分なユーザーベース（最低1,000 DAU推奨）

---

## A/Bテストの基本原則

### 1. テスト優先度マトリクス

| インパクト / 実装コスト | 低コスト | 高コスト |
|----------------------|---------|---------|
| **高インパクト** | ★★★ 最優先 | ★★ 検討 |
| **低インパクト** | ★ 余裕時 | ✕ 見送り |

### 2. 推奨テスト順序

| 優先度 | テスト領域 | 期待改善KPI | 難易度 |
|--------|-----------|-----------|--------|
| 1 | オンボーディング | リテンション +10-30% | 中 |
| 2 | ペイウォール | 課金CVR +20-50% | 低 |
| 3 | 通知タイミング | DAU +5-15% | 低 |
| 4 | 価格設定 | ARPU +10-30% | 低 |
| 5 | UI/UX | セッション時間 +5-20% | 中 |
| 6 | 広告配置 | 広告収益 +10-25% | 低 |

## Step 1: テスト設計

### 1.1 仮説設定フレームワーク

```markdown
## 仮説テンプレート

**現状の問題:**
[現在のメトリクスと課題を記述]

**仮説:**
[変更X]を行うことで、[指標Y]が[Z%]改善すると考える。

**根拠:**
- [データや過去の知見に基づく根拠1]
- [根拠2]

**リスク:**
- [考えられるネガティブ影響]
```

### 1.2 サンプルサイズ計算

| 現在CVR | 期待改善 | 必要サンプル（各群） | 95%信頼度 |
|---------|---------|-------------------|-----------|
| 1% | +20% (→1.2%) | 約65,000 | ✓ |
| 2% | +20% (→2.4%) | 約32,000 | ✓ |
| 5% | +20% (→6%) | 約12,000 | ✓ |
| 10% | +20% (→12%) | 約6,000 | ✓ |

**計算式:**
```
n = 2 × (Zα + Zβ)² × p(1-p) / δ²

Zα = 1.96 (95%信頼度)
Zβ = 0.84 (80%検出力)
p = 基準転換率
δ = 期待改善幅
```

### 1.3 テスト期間の決定

| DAU | 必要サンプル | 最小期間 | 推奨期間 |
|-----|------------|---------|---------|
| 1,000 | 12,000 | 12日 | 2週間 |
| 3,000 | 12,000 | 4日 | 1週間 |
| 10,000 | 12,000 | 2日 | 1週間 |

**注意:**
- 最低7日間は実施（曜日変動を考慮）
- イベント期間・祝日は避ける

## Step 2: テスト実装

### 2.1 Firebase Remote Config 設定

```dart
// Remote Config パラメータ例
{
  "onboarding_flow": "control", // control | variant_a | variant_b
  "paywall_design": "control",
  "notification_timing": "control",
  "ad_frequency": "control"
}
```

### 2.2 テストグループ割り当て

```dart
// ユーザーをランダムに振り分け
import 'package:firebase_remote_config/firebase_remote_config.dart';

class ABTestManager {
  final remoteConfig = FirebaseRemoteConfig.instance;

  Future<void> initialize() async {
    await remoteConfig.setConfigSettings(RemoteConfigSettings(
      fetchTimeout: const Duration(minutes: 1),
      minimumFetchInterval: const Duration(hours: 1),
    ));
    await remoteConfig.fetchAndActivate();
  }

  String getVariant(String testName) {
    return remoteConfig.getString(testName);
  }
}
```

### 2.3 イベントトラッキング設定

// テスト開始時のログ
await FirebaseAnalytics.instance.logEvent(

## Step 3: テスト領域別ガイド

### 3.1 オンボーディングテスト

| テスト項目 | Control | Variant案 | 測定指標 |
|-----------|---------|-----------|---------|
| ステップ数 | 5ステップ | 3ステップ | 完了率 |
| スキップ可否 | スキップ不可 | スキップ可能 | D1リテンション |
| パーソナライズ | なし | 質問→カスタマイズ | エンゲージメント |
| 価値訴求 | 機能説明 | ベネフィット訴求 | 課金CVR |

**成功基準例:**
- オンボーディング完了率: +15%
- D1リテンション: +5%

### 3.2 ペイウォールテスト

| テスト項目 | Control | Variant案 | 測定指標 |
|-----------|---------|-----------|---------|
| 表示タイミング | 起動時 | 機能利用後 | CVR |
| デザイン | シンプル | 比較表 | CVR |
| 価格表示 | 月額のみ | 月額+年額 | 年額比率 |
| 無料トライアル | 3日 | 7日 | CVR・継続率 |
| 割引訴求 | なし | 期間限定-20% | CVR |

**成功基準例:**
- 課金CVR: +25%
- 年額選択率: +10%

### 3.3 通知テスト

| テスト項目 | Control | Variant案 | 測定指標 |
|-----------|---------|-----------|---------|
| 送信時刻 | 固定（9:00） | パーソナライズ | 開封率 |
| 頻度 | 毎日 | 週3回 | オプトアウト率 |
| 文言 | 機能的 | 感情的 | タップ率 |
| リッチ通知 | テキストのみ | 画像付き | タップ率 |

**成功基準例:**
- 通知開封率: +20%
- 通知経由DAU: +10%

### 3.4 価格テスト

| テスト項目 | Control | Variant案 | 測定指標 |

## 🤖 Claude Code への指示

**トークン効率化:**
- テスト設計は `docs/project/ab_tests/` に保存
- Firebase Remote Config設定はコード生成
- 結果分析はBigQuery連携（設定済みの場合）

**実行モード:**
1. `--design [テスト名]`: 新規テスト設計
2. `--implement [テスト名]`: Remote Config + Analytics設定
3. `--analyze [テスト名]`: 結果分析・レポート生成
4. `--list`: 実行中・完了テスト一覧

**Firebase設定の自動化:**
- Remote Config パラメータ追加
- Analytics カスタムイベント定義
- BigQuery エクスポート設定（オプション）

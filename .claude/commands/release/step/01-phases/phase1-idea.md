# 🎯 Phase 1: アイデア探索

## 目的
収益性・差別化・リテンション・防御力の多軸評価で3案を選定

---

## 📥 入力モード判定

**最初に以下の順序で入力ソースを確認する。**

### モード1: 企画書モード（最優先）

```bash
# アプリ名を現在のディレクトリ名から取得
APP_NAME=$(basename $(pwd))
SPEC_PATH=~/kikiki/released/company/planning/specs/${APP_NAME}-spec.md
ls "$SPEC_PATH" 2>/dev/null
```

**企画書が存在する場合:**
1. 企画書のコンセプト・ターゲット・差別化を読み込む
2. **1案として確定**（3案生成は不要）
3. 7軸評価を実施（検証目的。企画書の判断を覆さない）
4. 企画書に記載のない項目のみ自動生成で補完

### モード2: 対話型入力モード

`docs/project/idea_input.md` が存在する場合:
1. `idea_input.md` の内容を読み込む
2. ユーザーのアイデアを**1案として確定**（3案生成は不要）
3. 7軸評価を実施し、弱点があれば改善提案
4. 「お任せ」項目は市場調査を基に最適化

### モード3: 自動探索モード

上記いずれも存在しない場合、以下の通常フローを実行（3案生成→選定）

## 🚫 制約条件（Pass/Fail）

| 制約 | 条件 |
|------|------|
| 重複回避 | `gh repo list crelve --limit 100` の全リポジトリとテーマ重複なし |
| センシティブ除外 | 医療診断・法律アドバイス・金融アドバイス等、プロのライセンスが必要な専門職行為を AI が代替するテーマは除外 |
| SNS除外 | ユーザー間投稿・シェア機能なし |
| toC特化 | B2B・業務用は除外 |
| 女性ターゲット | 主要ターゲットユーザーが女性であること |
| **ポートフォリオバランス** | 既存アプリと同じバリュー構造が続いている場合、スコアに補正を加える（下記参照） |

### 📊 ポートフォリオバランス補正（スコア調整）

**Step 0 でリポジトリ一覧取得後**、既存アプリのコアバリュー構造を確認し、候補アイデアのスコアを補正する。

**補正ルール:**

| 状況 | 補正 |
|------|------|
| 候補のコア動詞が「記録する」かつ既存に同構造が3本以上ある | -2点（差別化確実性軸） |
| 候補のターゲット層が既存アプリと完全一致（例：育児層が既に2本） | -1点（市場機会軸） |
| 既存にない新しいコア動詞（「計画する」「発見する」「学ぶ」等） | +1点（差別化確実性軸） |

**現在の構造分布（2026-03時点）:**
- `趣味記録 × AI生成`: SweetLog / EncoreLog / OshiNote / CoordNote（4本）
- `育児記録 × AI生成`: KidNote / PetalPages（2本）
- `生活記録 × AI分析`: LumiLog / CraftNest（2本）

→ **記録系が多い状況なので、「計画」「発見」「学習」軸のアイデアは素点が同じでも補正でリードしやすい。**
ただし記録系でも十分差別化されていて高スコアなら採用してよい。

### 重複チェック手順（必須）

**Step 0 として最初に実行すること：**

```bash
# crelve 組織の全リポジトリ一覧を取得
gh repo list crelve --limit 100 --json name,description
```

取得した全リポジトリのテーマ・コンセプトと候補アイデアを照合し、**類似するものを全て除外**する。
重複チェックのソースはこのコマンドのみを正とする（`project_history.json` は記録先であり確認ソースとして使わない）。

### センシティブテーマ 判定基準

以下に該当するテーマは **自動的に除外**する（App Store審査リスク・法的リスクのため）：

| カテゴリ | 具体例 | 理由 |
|---------|--------|------|
| 医療診断 | 症状トリアージ、病名推定、薬の用量判断 | 医師法・薬機法 |
| 獣医診断 | ペットの症状診断、緊急度評価 | 獣医師法 |
| 法律アドバイス | 契約書レビュー、法的判断 | 弁護士法 |
| 金融アドバイス | 投資推薦、ポートフォリオ管理 | 金融商品取引法 |
| 心理療法 | うつ・不安の診断・治療提案 | 医療行為 |

**境界ケースの判定**: AIが「あくまで参考情報」でも、緊急度評価・診断に相当する出力をする場合は除外。

## 📊 7軸評価（各5点・合計35点）

| 軸 | 5点 | 3点 | 1点 |
|----|-----|-----|-----|
| 💰 収益化確実性 | 安定したサブスク収益＋高LTV | 収益化の道筋明確で継続性あり | 不透明 |
| ⚡ 差別化明確性 | ブルーオーシャン | 差別化ポイント明確 | 困難 |
| 📊 市場機会 | SOM>$50K/年 | $10-50K/年 | <$10K/年 |
| ⚙️ 実現可能性 | 6週間以内でMVP可能 | 6-12週間 | 12週間以上 |
| 🔄 リテンション | 毎日使う必然性 | 週数回利用 | 単発利用 |
| 🛡️ 参入障壁 | 模倣困難 | 差別化維持可 | 簡単にコピー |
| 📢 バイラル適性 | 自然な口コミ発生 | オーガニック見込み | 広告依存 |

**足切り**: 収益化は最低**4点**必須・リテンション・市場機会は最低3点必須

## 探索プロセス

### Step 0: crelve全リポジトリ確認（最優先・必須）

```bash
gh repo list crelve --limit 100 --json name,description
```

- 取得した全リポジトリのテーマを記録する
- この一覧と重複するテーマは **候補から即除外**する
- `project_history.json` は補助情報として参照（全件は含まれない）

**サフィックス使用回数も同時に確認し、アイデア命名の方向性を決める:**
```bash
gh repo list crelve --limit 100 --json name --jq '.[].name' | \
  grep -oE '\-[a-z]+$' | sort | uniq -c | sort -rn | head -15
```
2回以上使われているサフィックス（-note, -log, -ai, -mind等）は **アイデアの仮名段階から回避**すること。Phase 2 でより厳格にチェックする。

### Step 1-2: 制約チェック → アイデア発想
- Step 0 の除外リストを適用してセンシティブテーマもフィルタリング
- SCAMPER法で既存アプリを変形（除外済みテーマを避けて発想）

### Step 3: 市場規模推定（グローバル基準）

**対象市場**: グローバル（US を主要ターゲット）

```
TAM = グローバルカテゴリ市場規模
SAM = TAM × ターゲット比率 × 対応言語圏比率
SOM = SAM × シェア目標(0.1-1%)
```

**市場規模参考データソース:**
- Sensor Tower / data.ai（旧App Annie）
- Statista Mobile App Market
- Grand View Research

**地域別モバイルアプリ市場シェア（参考）:**
| 地域 | シェア |
|------|--------|
| 🇺🇸 North America | 30-35% |
| 🇪🇺 Europe | 20-25% |
| 🇯🇵🇰🇷 Japan/Korea | 10-12% |
| 🇨🇳 China | 15-20% |
| 🌏 Rest of World | 15-20% |

### Step 4-6: 4象限配置 → 総合評価 → 選定

| スコア | 判定 |
|--------|------|
| 28-35点 | ★★★ 即採用 |
| 21-27点 | ★★ 詳細検討 |
| 14-20点 | ★ 要改善 |
| 0-13点 | 見送り |
